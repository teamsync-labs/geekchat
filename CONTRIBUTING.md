# Участие в разработке GeekChat

Спасибо, что хотите помочь с проектом.

## Перед началом

1. Ознакомьтесь с [vision](docs/vision.md).
2. Согласуйте задачу с командой, чтобы не дублировать работу.

## Стиль кода

- Следуйте соглашениям, уже принятым в соответствующей папке.
- Не смешивайте несвязанные изменения в одном PR.

## Вопросы

Если что-то неясно — спросите в общем чате команды или в комментариях к issue/PR.

## Архитектура

Backend и веб-клиент:

- **Backend** (`backend/`) — API, комнаты, WebSocket signaling, авторизация.
- **Frontend** (`frontend/`) — веб-клиент: лендинг, UI звонка, WebRTC.

Видео и аудио идут напрямую между браузерами (P2P); backend не передаёт медиапоток.

## Где что лежит

| Область | Папка | Кто обычно работает |
| --- | --- | --- |
| API, signaling, авторизация | `backend/` | Python-разработчик |
| Веб-клиент | `frontend/` | Frontend-разработчик |
| Документация, PRD, макеты | `docs/` | Продуктовый дизайнер / PM |
| Проверки, тест-кейсы | `docs/`, карточки задач | QA |
| Инфраструктура | `infra/` | По договорённости |

Таблица показывает, к кому обычно идти с вопросами — не запрещает работать в других папках.

## Быстрый старт (первый PR)

Подробности — в разделе «Полный процесс» ниже.

1. Возьмите задачу из **To Do** и перенесите в **In Progress**.
2. Создайте ветку от `main`: `feature/краткое-описание` или `fix/краткое-описание`.
3. Делайте коммиты в формате `feat: ...`, `fix: ...` (другие типы — в п. 3.2).
4. Откройте pull request в `main` со ссылкой на карточку задачи.
5. Перенесите карточку в **Ready for Review**.
6. Берите следующую задачу — merge в `main` выполняет ревьюер после проверки.

## Полный процесс разработки: задачи и Git

### 1. Базовые правила

1.0. Наша общая цель — уменьшать количество несделанной работы.
Это значит, что в первую очередь мы доводим до ума те задачи, которые ближе к завершению на доске задач.
В In Progress стараемся держать только одну свою задачу - так мы знаем кто чем занят и можем прогнозировать сроки выполнения задач.

1.1. У задачи в статусе In Progress должен быть назначен исполнитель.

1.2. Пока проект не развернут на сервере, используем одну стабильную ветку: `main`.

1.3. Единый путь разработки: `main -> новая ветка -> pull request -> main`.

1.4. Освободившийся разработчик берет в работу следующую задачу из назначенных ему в статусе To Do.

1.5. Если в To Do нет задач, разработчик уточняет в чате команды, что делать дальше.

1.6. Разработчик не берет в работу задачи, заблокированные лейблом `Требует уточнения` или если в задаче явно указана зависимость от задачи, которая еще не выполнена.

### 2. Ветки

2.1. Каждую новую задачу делаем в новой ветке.

2.2. Формат имени ветки: `feature/краткое-описание` или `fix/краткое-описание` (например, `feature/post-filters`).

2.3. Новые ветки создаем от `main`:

```bash
git checkout main && git pull origin main && git checkout -b <my-feature-branch>
```

2.4. В карточке задачи указываем название новой ветки.

### 3. Коммиты

3.1. Формат commit message: `<type>: <краткое описание>`.

3.2. Допустимые `type`: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`.

3.3. Force push (`--force`, `--force-with-lease`) не используем, если не понимаем зачем.

### 4. Перед переносом в Ready for Review

4.1. Подтягиваем изменения из `main` в свою ветку:

```bash
git checkout <my-feature-branch> && git pull origin main
```

4.2. Пушим свою ветку: `git push origin <my-feature-branch>`.

4.3. В GitHub создаем pull request (PR) в ветку `main`. В заголовке PR пишем название ветки или название задачи.

4.4. В pull request добавляем ссылку на карточку задачи.

4.5. В карточку задачи добавляем ссылку на pull request.

4.6. В карточку задачи добавляем описание минимальных проверок в браузере.

4.7. Если минимальные проверки не проходят у автора - не отдаем на проверку.

4.8. Перед переносом в Ready for Review старайтесь устранить конфликты с `main` самостоятельно (п. 4.1). Если это вызывает затруднения — оставьте комментарий в карточке задачи; ревьюер разрешит конфликты.

4.9. Переносим карточку задачи в колонку Ready for Review.

### 5. Статус Testing

5.0. В Testing задачу переносит ревьюер.

5.1. Когда задача в статусе Testing и у ревьюера нет вопросов — не добавляйте новые коммиты, даже если GitHub сообщил о появившихся конфликтах.
Такое случается, когда команда вносит изменения один и тот же фрагмент кода, работая над близкими по смыслу задачами. В этом случае ревьюер разрешит конфликты самостоятельно.

5.2. Если потребуется что-то исправить — см. п. 6.1. При срочности ревьюер свяжется с вами напрямую.

### 6. Итоги проверки ревьюером

6.1. Если в задаче что-то не доделано или требуются уточнения — ревьюер добавит комментарий в карточку, карточка перейдёт в статус Need to Fix. Доработайте по комментариям и снова переведите задачу в Ready for Review.

6.2. Если задача выполнена — карточка переходит в статус Done; pull request одобряет ревьюер — ваша работа сливается в ветку `main`.

6.3. Перед доработкой задачи из Need to Fix, перенесите ее в In Progress. Так команда будет видеть, что реально в работе.

6.4. Всегда думаем о пункте 1.0 и обрабатываем свои карточки в таком порядке (справа налево): Need to Fix -> Ready for Review (если вы ревьюер) -> In Progress -> To Do.
