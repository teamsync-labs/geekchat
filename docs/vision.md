# Что мы строим

GeekChat — открытый учебный проект по разработке веб-приложения.

Наша цель — не сделать очередной аналог Zoom.

Наша цель — научиться создавать продукты в команде, используя реальные инженерные практики.

После завершения проекта каждый участник сможет показать работодателю:

- публичный GitHub-репозиторий;
- свои Pull Request;
- историю коммитов;
- участие в Code Review;
- работающий продукт;
- страницу команды;
- сертификат участия;
- рекомендательное письмо.

# Что считается успешным MVP

Через 3–6 недель должно работать следующее.

Пользователь А

↓


получает ссылку

↓

отправляет ссылку пользователю Б

↓

пользователь Б открывает ссылку

↓

начинается видеозвонок.

Если этот сценарий работает стабильно —

MVP считается завершенным.

# Наши принципы

Мы пришли сюда не просто написать код.

Мы учимся работать как инженерная команда.

Поэтому придерживаемся следующих правил.

- Все изменения проходят через систему контроля версий.
- Работа ведётся небольшими Pull Request.
- Каждый Pull Request проходит Code Review.
- Архитектурные решения обсуждаются открыто.
- Ошибаться нормально.
- Не задавать вопросы — хуже, чем задать "глупый" вопрос.
- Документация развивается вместе с проектом.
- Любой участник может предложить улучшение продукта или процесса.

# Архитектура MVP

```
Browser
      │
      │ HTTPS / WebSocket
      ▼
 FastAPI
      │
      ▼
 PostgreSQL

Browser ←──── WebRTC P2P ────→ Browser
```

- авторизацию;
- создание комнат;

## Backend Python

Какие архитектурные решения стоит принять сразу?

1. Аутентификация (JWT)

Правильный поток:

1. Логин → выдача access JWT (короткий, 10–30 мин)
2. Refresh токен хранится отдельно
3. WebSocket подключается только с валидным JWT

В FastAPI:
- проверка токена через Depends
- подпись через HS256 или лучше RS256
- секрет хранить в env

Важно:
— не хранить JWT в localStorage (лучше httpOnly cookie)
— использовать HTTPS обязательно

2. Безопасность WebSocket

Основные риски — подключение без авторизации и спам.

Что сделать:

✅ Проверять JWT при подключении
✅ Привязывать connection к user_id
✅ Ограничить частоту сообщений (rate limit)
✅ Проверять, что пользователь имеет доступ к комнате

3. Комнаты и доступ

Не доверять client-side данным.
Можно добавить временные invite-токены.

Никогда:
❌ не передавать просто room_id без проверки

Всегда:
✔️ генерировать UUID для комнат
✔️ проверять в БД, что пользователь участник комнаты


4. PostgreSQL безопасность

Минимум:

✅ отдельный пользователь БД
✅ не использовать superuser
✅ индексы на user_id / room_id
✅ миграции через Alembic

Пароли:
— только bcrypt или argon2
— никогда не хранить plain text

5. STUN / TURN

Для продакшена нужен TURN.

Важно:
— TURN с авторизацией
— временные credentials
— не публичный open relay

Иначе сервер будут использовать как прокси.

6. Дополнительная защита API

✔️ CORS настроен строго на доменный адрес проекта
✔️ Rate limit на login
✔️ Защита от brute force
✔️ Логирование подозрительных попыток
✔️ Docker + firewall

## QA

Как понять, что продукт действительно готов? 
-> Нет блокирующих и критичных багов
-> Продукт выполняет бизнес-цель: "создать ссылку → отправить → начать видеозвонок"
-> Проект соответствует макетам и ведёт себя в соответствии с требованиями (негативный сценарий - ничего «не падает»)


Какие проверки обязательно должны быть автоматизированы?
Пока вопрос. Возможно автоматизация формы регистрации, но её целесообразно автоматизировать в будущем, когда планируется подключение большого количества человек.

# Definition of Done

MVP считается завершенным, если:

- работает сценарий "создать ссылку → отправить → начать видеозвонок";
- проект доступен по публичному адресу;
- код находится в открытом GitHub;
- все основные изменения прошли Code Review;
- документация актуальна;
- каждый участник может показать свой вклад работодателю.
